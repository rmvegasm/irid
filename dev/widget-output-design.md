# WidgetOutput ‚Äî Design Document

**Status:** Draft  
**Date:** May 2026

---

## 1. Motivation

`Output()` and its wrappers (`PlotOutput`, `TableOutput`, `DTOutput`) make it
easy to embed any Shiny render/output pair in an irid tag tree. htmlwidgets ‚Äî
the ecosystem of JavaScript visualization libraries wrapped for Shiny ‚Äî work
with `Output()` out of the box for the rendering half. But they also produce
Shiny inputs. A plotly chart fires `{id}_click`; a leaflet map fires
`{id}_shape_click`; a mapgl map fires `{id}_feature_click`. These inputs are how
the widget exposes user interaction back to R.

`Output()` has no mechanism for widget inputs. A user who needs them today must
reach around the framework:

```r
session <- shiny::getDefaultReactiveDomain()
observe({
  click <- session$input[["irid-7_feature_click"]]  # hardcoded auto-generated ID
  selected(click$feature)
})
```

This breaks irid's contract: the user must know the auto-generated element ID,
hold a session reference outside the tag tree, and wire observers by hand. There
is no principled way to scope or tear down these observers when the widget's
containing `When`/`Each` branch is destroyed.

A second gap: many widgets ‚Äî leaflet, mapgl, DT, and others ‚Äî provide **proxy
functions** that update an already-rendered widget without a full re-render
(e.g. `leaflet::leafletProxy()`, `mapgl::maplibre_proxy()`). These proxies also
require the widget's output ID, which is not accessible to the user when the ID
is auto-generated by irid.

`WidgetOutput` closes both gaps. It accepts a render/output pair and a function
‚Äî identical to `Output()` ‚Äî plus:

- **`update`**: an optional reactive update function that receives the
  auto-generated ID and is mounted as an observer. The user calls the widget's
  proxy function inside it to apply R-driven changes incrementally.
- **Named event receptors**: named callables that are called when the widget
  fires the corresponding Shiny input. irid wires up the `observeEvent` calls
  internally and tears them down with the rest of the mount.

---

## 2. Goals

- Same authoring pattern as `Output()`: `render_fn`, `output_fn`, a function,
  it's reactive
- Optional `update` function gives the user access to the auto-generated ID for
  proxy-based incremental updates
- Widget Shiny inputs are bound to user-provided callables via named
  event-receptor arguments
- Event receptors and the update function live side-by-side, consistent with
  `PlotlyOutput`'s API shape
- All observers are scoped to the mount and destroyed when the widget is torn
  down
- No widget-specific knowledge required ‚Äî works with any htmlwidgets-compliant
  widget that follows the `{outputId}_{suffix}` naming convention
- No JS additions ‚Äî all wiring is R-side `observe` / `observeEvent` calls

---

## 3. Design Principles

### Two mechanisms, two directions

`WidgetOutput` separates the two directions of state flow into two distinct
mechanisms:

**R ‚Üí widget: the `update` function.**  An optional reactive function that
receives the auto-generated output ID. irid mounts it as an `observe()`, so any
reactive reads inside it create the usual reactive dependencies. The user calls
the widget's proxy function inside to apply incremental updates without a full
re-render:

```r
update = \(id) {
  mapgl::maplibre_proxy(id) |>
    mapgl::set_center(center()) |>
    mapgl::set_zoom(zoom())
}
```

**Widget ‚Üí R: named event receptors.**  Named callable arguments (non-`update`,
non-`on*`) that observe the widget's Shiny inputs. When the widget fires
`Shiny.setInputValue('{id}_center', ...)`, irid calls the bound callable with
the new value:

```r
center = center_rv,
zoom   = zoom_rv
```

Together, these two mechanisms give `WidgetOutput` the same bidirectional data
flow as `PlotlyOutput` ‚Äî without requiring irid to own the widget's JS rendering
path.

### How this compares to `PlotlyOutput`

`PlotlyOutput` owns its entire JS rendering path via `Plotly.react()`. It can
apply state incremetally from R (state fields are one-way-in bindings, R ‚Üí
widget) and capture user interactions via explicit `on*` callbacks (widget ‚Üí R).
The user never calls a proxy ‚Äî irid handles it.

`WidgetOutput` delegates rendering to htmlwidgets and does not own the JS path.
The user is responsible for calling the widget's proxy in the `update` function.
In exchange, `WidgetOutput` works generically across any proxy-capable widget.
The mental model is:

- `PlotlyOutput`: irid manages the rendering path, you declare state and
  callbacks
- `WidgetOutput`: you manage the rendering path (via the proxy), irid handles
  scoping and teardown

For widgets without proxy support (simple tables, static charts), `update` is
omitted and `WidgetOutput` behaves like `Output()` with event receptors.
R-driven changes to widget state require a full re-render; see Section 5.

### No universal widget state constructor

There is no `widget_state()` equivalent ‚Äî each widget defines its own Shiny
inputs and there is no canonical widget state shape. Users construct their own
stores with exactly the fields they need, following the `plotly_state()`
pattern:

```r
map_state <- reactiveStore(list(center = NULL, zoom = NULL))
```

Widget-specific packages may build their own state constructors and sync helpers
on top of `WidgetOutput`, following the `plotly_state()` / `plotly_sync()`
pattern.

### Relationship to `Output()`

When called without `update` and without named event receptors, `WidgetOutput`
is functionally identical to `Output()`:

```r
# These are equivalent
Output(mapgl::renderMaplibre, mapgl::maplibreOutput, fn)
WidgetOutput(mapgl::renderMaplibre, mapgl::maplibreOutput, fn)
```

## 4. API

### Basic usage ‚Äî render only

```r
WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() create_maplibre_instance(center = c(-74, 40), zoom = 10)
)
```

Identical to `Output(mapgl::renderMaplibre, mapgl::maplibreOutput, fn)`.
Additional positional arguments after `fn` are forwarded to the output function.

### Proxy-based incremental updates

The `update` function receives the widget's auto-generated output ID and runs
reactively as an observer. Any reactive values read inside it (here `center()`
and `zoom()`) invalidate and re-run the update when they change, calling the
proxy to apply changes incrementally:

```r
center <- reactiveVal(c(-74, 40))
zoom   <- reactiveVal(10)

WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() create_maplibre_instance(center = c(-74, 40), zoom = 10),
  update = \(id) {
    mapgl::maplibre_proxy(id) |>
      mapgl::set_center(center()) |>
      mapgl::set_zoom(zoom())
  }
)
```

The render function sets the initial state. Subsequent R-driven changes go
through the `update` observer ‚Äî no full re-render.

### Event receptors ‚Äî widget ‚Üí R

Named callable arguments observe the widget's Shiny inputs. When the widget
fires `{id}_center`, irid calls `center(value)`:

```r
center <- reactiveVal(NULL)
zoom   <- reactiveVal(NULL)

WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() create_maplibre_instance(center = c(-74, 40), zoom = 10),
  center = center,
  zoom   = zoom
)
```

### Full bidirectional pattern

Combining `update` and event receptors achieves the complete round-trip: R
drives the widget via proxy, user interactions write back via event receptors.

```r
center <- reactiveVal(c(-74, 40))
zoom   <- reactiveVal(10)

WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() create_maplibre_instance(center = c(-74, 40), zoom = 10),
  update = \(id) {
    mapgl::maplibre_proxy(id) |>
      mapgl::set_center(center()) |>
      mapgl::set_zoom(zoom())
  },
  center = center,
  zoom   = zoom
)

# A "fly to Boston" button updates the reactives; the update observer
# fires and calls the proxy ‚Äî no re-render, no loop.
tags$button(
  "Fly to Boston",
  onClick = \() { center(c(-71.06, 42.36)); zoom(12) }
)
```

The reactive loop risk (user pans ‚Üí receptor writes ‚Üí update fires ‚Üí proxy sets
position ‚Üí widget fires again) does not arise in practice as long as the widget
only fires Shiny inputs in response to user interaction, not in response to
programmatic proxy calls. Well-designed widgets (leaflet, mapgl) follow this
convention.

### Using a store for bookmarking

```r
map_state <- reactiveStore(list(center = NULL, zoom = NULL))

WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() {
    s <- isolate(map_state())
    create_maplibre_instance(
      center = s$center %||% c(-74, 40),
      zoom   = s$zoom   %||% 10
    )
  },
  update = \(id) {
    mapgl::maplibre_proxy(id) |>
      mapgl::set_center(map_state$center()) |>
      mapgl::set_zoom(map_state$zoom())
  },
  center = map_state$center,
  zoom   = map_state$zoom
)

# Serialize for bookmark
saveRDS(map_state(), "bookmark.rds")

# Rehydrate
map_state <- reactiveStore(readRDS("bookmark.rds"))
```

`isolate()` in the render function prevents the initial store read from creating
a reactive dependency ‚Äî the render function should run once, with `update`
handling all subsequent changes. The bookmark pattern mirrors `plotly_state()` /
`plotly_sync()` and composes the same way.

### Action callbacks

For widget events that carry no persistent state ‚Äî a feature click that
populates a detail panel, a selection that filters another chart ‚Äî pass a
callback as a named argument alongside state receptors. See Open Questions (¬ß8)
for the naming convention decision.

```r
selected <- reactiveVal(NULL)

WidgetOutput(
  mapgl::renderMaplibre,
  mapgl::maplibreOutput,
  \() create_maplibre_instance(...),
  center = center,
  zoom = zoom,
  onFeature_click = \(event) selected(event$feature)
)
```

## 5. Widgets Without Proxy Support

For widgets that do not provide a proxy function, the `update` argument is
omitted and R-driven state changes require a full re-render. This is identical
to standard Shiny widget behavior and acceptable for widgets where re-render
cost is low (tables, simple charts).

If the render function reads a reactive that is also an event receptor, every
user interaction triggers a re-render. To avoid this while retaining R-driven
updates, use `isolate()` in the render function for event-receptor reactives ‚Äî
the tradeoff is that R-driven changes to those values won't trigger a re-render
either:

```r
# isolate() breaks the reactive dependency in the render function.
# R-driven changes to `zoom` won't trigger a re-render; the widget
# only re-renders when other reactive data (e.g., the dataset) changes.
WidgetOutput(
  some::renderWidget,
  some::widgetOutput,
  \() create_widget(zoom = isolate(zoom())),
  zoom = zoom
)
```

This is a per-widget tradeoff between re-render cost and R-driven state
fidelity. There is no generic answer ‚Äî document the choice in the component's
comment or README.

## 6. Relationship to PlotlyOutput

Plotly is an htmlwidgets-compliant widget. `PlotlyOutput` is not built on
`WidgetOutput` ‚Äî it is a separate, more capable primitive that owns its JS
rendering path via `Plotly.react()`.

| | `WidgetOutput` (with proxy) | `WidgetOutput` (no proxy) | `PlotlyOutput` |
|---|---|---|---|
| R ‚Üí widget updates | Via user-provided proxy in `update` | Full re-render | Incremental via `Plotly.react()` |
| Widget ‚Üí R events | Named event receptors | Named event receptors | `on*` callbacks |
| JS additions | None | None | Custom `irid-plotly-render` handler |
| User controls rendering path | Yes | Yes | No ‚Äî irid controls it |

The rule: if a widget exposes a JS API for incremental updates, use
`WidgetOutput` with an `update` function. If a widget requires irid to own the
entire rendering path (e.g. to merge state into a spec before sending), write a
dedicated output type like `PlotlyOutput`. `WidgetOutput` with proxy covers the
majority of interactive htmlwidgets.

## 7. Implementation

### R side

`WidgetOutput` returns an `irid_widget_output` node ‚Äî a new S3 class separate
from `irid_output`. It carries:

- `output_fn`, `output_fn_args`, `render_call` ‚Äî same as `irid_output`
- `update_fn` ‚Äî the `update` argument (or `NULL`)
- `state_fields` ‚Äî named list of callables (non-`update`, non-`on*` named args)
- `callbacks` ‚Äî named list of functions (`on*` named args)

Argument classification in `WidgetOutput()` itself: the `update` argument is
extracted by name; any remaining named argument starting with `on` followed by
an uppercase letter is a callback; all other named arguments are state fields;
unnamed arguments after `fn` go to `output_fn_args`.

`process_tags` handles `irid_widget_output` nodes identically to `irid_output`:
generate an ID, store in `shiny_outputs`, return `do.call(output_fn, c(list(id),
output_fn_args))`.

`irid_mount_processed` gains a mounting step for widget outputs:

```r
# Render call ‚Äî same as irid_output
session$output[[id]] <- node$render_call

# Update observer
if (!is.null(node$update_fn)) {
  local({
    ufn <- node$update_fn
    wid <- id
    obs <- observe({ ufn(wid) })
    observers[[length(observers) + 1L]] <<- obs
  })
}

# State field receptors
for (nm in names(node$state_fields)) {
  callable   <- node$state_fields[[nm]]
  input_name <- paste0(id, "_", nm)
  local({
    iname <- input_name
    cb    <- callable
    obs   <- observeEvent(
      session$input[[iname]],
      { cb(session$input[[iname]]) },
      ignoreInit = TRUE,
      ignoreNULL = TRUE
    )
    observers[[length(observers) + 1L]] <<- obs
  })
}

# Action callbacks
for (nm in names(node$callbacks)) {
  fn         <- node$callbacks[[nm]]
  suffix     <- on_name_to_suffix(nm)   # see ¬ß8
  input_name <- paste0(id, "_", suffix)
  local({
    iname   <- input_name
    handler <- fn
    obs     <- observeEvent(
      session$input[[iname]],
      { handler(session$input[[iname]]) },
      ignoreInit = TRUE,
      ignoreNULL = TRUE
    )
    observers[[length(observers) + 1L]] <<- obs
  })
}
```

The `local()` wrapper is required in all loops ‚Äî the for-loop closure captures
loop variables by reference.

All observers join the mount's `observers` list and are destroyed by
`mount$destroy()`.

### No JS side changes

All wiring is R-side `observe` / `observeEvent`. The widget's existing JS fires
`Shiny.setInputValue` as it already does in standard Shiny; proxy functions send
their own Shiny messages to the client. irid adds no client-side code for
`WidgetOutput`.

## 8. Open Questions

### Callback naming convention (`on*` ‚Üí input suffix)

State fields use the widget's input suffix verbatim: `center = rv` observes
`{id}_center`. For action callbacks, irid's existing convention is `on` +
PascalCase. Applied to widget input suffixes, the mapping is: strip `on`,
convert PascalCase to snake_case.

- `onClick` ‚Üí `click`
- `onZoom` ‚Üí `zoom`
- `onFeatureClick` ‚Üí `feature_click`

The PascalCase ‚Üí snake_case conversion is lossy for widgets that use camelCase
suffixes (`featureClick` not `feature_click`). Three options:

**Option A ‚Äî camelCase with documented limitation.** `onFeatureClick` ‚Üí
`feature_click`. Works for snake_case suffixes (the majority). Users with
camelCase suffixes fall back to a state field: `featureClick = \(e) inspect(e)`
(if plain functions are allowed as state field values ‚Äî see below).

**Option B ‚Äî drop `on*` callbacks entirely.** All widget events use the
state-field syntax. Fire-and-forget cases compose via a `reactiveVal` +
`observe`:

```r
last_click <- reactiveVal(NULL)

WidgetOutput(..., feature_click = last_click)

observe({
  click <- last_click()
  if (!is.null(click)) inspect(click)
})
```

Eliminates the naming ambiguity at the cost of minor verbosity for one-shot
handlers.

**Option C ‚Äî `on_snake_case` prefix for callbacks.** `on_feature_click`,
`on_zoom` ‚Äî deviate from irid's `onCamelCase` convention but match widget
suffixes verbatim with a disambiguating prefix. Requires detecting the `on_`
prefix as distinct from state field names.

Recommendation: ship with Option B (state fields only). Promote to Option A if
usage patterns show inline callbacks are commonly needed.

### `ignoreNULL` behavior for event receptors

`observeEvent(..., ignoreNULL = TRUE)` (the default) prevents the observer from
firing when the input is `NULL`. Widget inputs are typically `NULL` before the
first user interaction, so this is usually correct ‚Äî the callable keeps its own
initial value. However, `ignoreNULL = TRUE` means a widget cannot explicitly
signal "no selection" by setting its input to `NULL`.

Default: `TRUE`. Document the limitation. A future per-field `ignoreNULL` option
can be added if explicit-`NULL` signals prove common.

### Reactive loop risk in the update observer

The full bidirectional pattern ‚Äî event receptor writes a reactive, `update`
observer reads it and calls the proxy ‚Äî is only loop-free if the widget does not
fire Shiny inputs in response to programmatic proxy calls. Well-designed widgets
(leaflet, mapgl) only fire inputs on user interaction. Widgets that fire on
programmatic updates would produce a loop; this should be flagged in
documentation for affected widgets.

### Widgets that don't follow `{outputId}_{suffix}` convention

Some widgets use custom input IDs not derived from the element ID, or use
dot-separated paths (`{id}.zoom`). `WidgetOutput` cannot address these cases ‚Äî
users must fall back to `session$input` directly. Document as an explicit
limitation.

Alternatively, a more agressive approach using `snakecase::to_any_case` could be
implemented.

### Lifecycle on dynamic remount

When `WidgetOutput` lives inside a `When` or `Each` branch that is torn down and
remounted, all observers (including `update`) are destroyed and recreated, and
`session$output[[id]]` is reassigned. Widget-specific cleanup (layer disposal,
event listener removal) is the widget's responsibility via its htmlwidgets
cleanup hooks.
